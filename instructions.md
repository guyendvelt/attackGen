# AttackGen — Red Team Instructions

> Project guide for the **AI Red Team vs. Blue Team Bootcamp – Build Challenge**.
> This document explains *what* we are building and *how the pieces connect*. It is not the implementation.

---

## 1. Project Overview

**AttackGen** is the Red Team's command-line telemetry generator.

Its job is to generate **realistic process-command datasets on demand**, based on an attack scenario chosen at demo time (for example: ransomware, crypto miner, lateral movement, persistence, privilege escalation, or data exfiltration).

The tool produces a labeled dataset that mixes a small number of malicious process commands into a large body of benign operational commands. The Blue Team then tries to detect the malicious rows. Everything we generate is **text telemetry only** — strings that look like real command-line activity. Nothing is ever executed.

The dataset is **generated live by the tool** from the judges' scenario. We do not ship a fixed, hand-written CSV.

---

## 2. Challenge Requirements

These requirements are strict. The generator must meet them exactly.

- **~220 total rows** in the dataset.
- **Exactly 20 malicious rows.**
- **~200 benign rows.**
- The CSV must have **exactly these columns, in this order**:
  - `process_name`
  - `command_line`
  - `label`
- The `label` value must be **either `benign` or `malicious`** — nothing else.
- Commands must be **realistic process commands**, not placeholders or obvious dummy text.
- The dataset must be **generated by the tool** from the given scenario — not manually authored as a static CSV.
- A **separate malicious ground-truth file** must be produced, listing the 20 malicious commands.
- A **written attack story** must be produced, explaining the simulated attack.

---

## 3. Project Scope and Safety Boundaries

AttackGen generates **text data only**. It is a telemetry simulator, not an attack tool.

The tool **must not**:

- Execute, spawn, or shell out to any command it generates.
- Perform any real attack against any system, network, or service.
- Use real credentials, tokens, API keys, secrets, customer data, or private infrastructure.
- Reference real hostnames, internal IPs, accounts, or environments that exist outside this challenge.

Any "suspicious" or "malicious" commands in the output exist **purely as simulated telemetry strings** for the detection exercise. They are labels on text, not actions. If a command looks dangerous, it is still only a string in a CSV.

**Scope is process-level activity only.** No real network attacks, no real infrastructure attacks, no real malicious execution.

## 4. Team Responsibilities

The work is split into three independent workstreams. Each owner has clear inputs and outputs so the parts plug together cleanly.

### 4.1 UI Owner

- **Owns:** the interface that collects the request from the user/judge.
- **Inputs (from user):** attack scenario, selected categories, command counts, OS profile, and optionally a stealth level.
- **Outputs (to generator):** a structured request — scenario/story context, OS profile, a dictionary of benign category counts (summing to 200), and a dictionary of malicious category counts (summing to 20).
- **Not responsible for:** how commands are stored, how the CSV is composed, or validation of final totals. The UI only gathers and forwards the request.

### 4.2 Command Database Owner

- **Owns:** the pool of command templates and examples that everything draws from.
- **Inputs:** the agreed category list and the shared command data model.
- **Outputs:** a large, well-tagged pool of command objects covering every benign and malicious category, with enough entries per category to satisfy any reasonable request.
- **Not responsible for:** UI input handling or final CSV assembly. The database just needs to be rich, realistic, and correctly tagged.

### 4.3 CSV Generator and Dataset Composer Owner (Yoav)

- **Owns:** the dataset composition and export.
- **Inputs:** selected categories and per-category counts (benign and malicious), the OS profile, and access to the command database.
- **Outputs:** the final output files (see §7) — labeled CSV, unlabeled CSV, ground-truth file, attack story, optional summary.
- **Responsibilities:**
  - Pull matching commands from the database for each requested category.
  - Validate the final split (see §8).
  - Blend and shuffle benign and malicious rows so malicious rows are not clustered.
  - Write the output files with the exact required format.
- **Not responsible for:** authoring command content (that's the database owner) or collecting user input (that's the UI owner).

## 5. Shared Data Model

Internally, every command is represented as a **command object** with these fields:

| Field | Description |
|-------|-------------|
| `process_name` | The process/binary name (e.g. `bash`, `python3`, `powershell.exe`). |
| `command_line` | The full command-line string as it would appear in telemetry. |
| `label` | `benign` or `malicious`. |
| `category` | The generation category (see §10), e.g. `devops`, `persistence`. |
| `os_profile` | Target OS profile, e.g. `linux`, `windows`. |
| `scenario_tags` | Optional tags linking the command to scenarios (e.g. `ransomware`, `lateral_movement`). |
| `stealth_level` / `weight` | Optional. Relative subtlety or selection weight, used to bias toward realistic, less-obvious commands. |

**Export rule:** the final CSV must include **only** these three columns:

- `process_name`
- `command_line`
- `label`

All other fields (`category`, `os_profile`, `scenario_tags`, `stealth_level`/`weight`) are **internal only**. They guide selection and composition but never appear in `attack_dataset_labeled.csv`.

---

## 6. Category-Based Generation Flow

The end-to-end flow at demo time:

1. The user/judge **selects or enters an attack scenario** (e.g. ransomware).
2. The **UI decides or receives category counts** — how many commands to pull from each benign and malicious category.
3. The **CSV generator receives** the benign categories and malicious categories, each with counts, plus the OS profile and scenario context.
4. The **generator pulls commands** from the database for each requested category.
5. The **generator validates exact totals** (200 benign, 20 malicious, 220 total — see §8).
6. The **generator blends** benign and malicious commands and shuffles row order.
7. The **generator writes the final output files** (see §7).

---

## 7. Required Output Files

| File | Purpose |
|------|---------|
| `attack_dataset_labeled.csv` | The main deliverable. ~220 rows with columns `process_name`, `command_line`, `label`. Contains exactly 20 malicious and ~200 benign rows. |
| `attack_dataset_unlabeled.csv` | Same rows as the labeled set but **without** the `label` column (only `process_name`, `command_line`). This is what the Blue Team detector consumes. |
| `ground_truth_malicious.csv` | The 20 malicious commands only, as the answer key for scoring. |
| `attack_story.md` | A short written narrative explaining the simulated attack scenario — what the attacker did, in what order, and how the malicious commands map to that story. |
| `summary.json` *(optional)* | Machine-readable run summary: scenario, OS profile, per-category counts, totals, and any seed/config used. Useful for reproducibility and debugging. |

---

## 8. Validation Rules

Before writing output, the generator **must validate**:

- **Exactly 20 malicious rows.**
- **Exactly 200 benign rows.**
- **Exactly 220 total rows.**
- **No missing `process_name`** on any row.
- **No missing `command_line`** on any row.
- **No invalid labels** — every `label` is exactly `benign` or `malicious`.
- **No extra columns** in the final CSV beyond `process_name`, `command_line`, `label` (internal-only fields must be stripped from the export).
- **Sufficient inventory:** the database has enough commands in every requested category to satisfy the requested counts. If a category is short, fail loudly rather than silently under-filling.

If any check fails, the generator should stop and report the problem clearly rather than emit a malformed dataset.

---

## 9. Attack Realism Strategy

The goal is to make detection **genuinely challenging without breaking realism**.

- **Malicious commands should not be cartoonishly obvious.** Avoid joke payloads or commands that scream "I am the attack." They should look like plausible activity that blends into operational noise.
- **Benign commands should be rich and varied**, covering realistic admin, developer, logging, backup, shell, package-management, container, and service-management activity.
- **Blend, don't separate.** Malicious rows must be shuffled throughout the benign rows, not grouped together.
- Use stealth level / weighting to favor subtle malicious commands that resemble legitimate operations (e.g. living-off-the-land style activity) over blatant ones.

The detector should have to work for it — but every row, benign or malicious, must still look like something a real operator or attacker could plausibly run.

## 11. Supported Attack Categories and Benign Lookalikes

We now have an official initial list of supported Red Team attack categories. **Every attack category must include both:**

1. **Malicious process-command behavior** — the simulated attacker activity for that category.
2. **Realistic benign lookalike behavior** — legitimate operational commands that resemble the malicious activity closely enough to make it blend in.

The goal is to produce process-command datasets where **malicious commands are hidden inside legitimate operational noise**. The tool should *not* generate random, obviously-malicious attack strings. It should generate realistic command-line telemetry where the malicious story is coherent, but the surrounding benign activity makes automated detection harder.

> **Safety boundary (applies to every category below):** all categories are for **simulated process-command telemetry only**. The project must not execute commands, perform real exploitation, access real credentials, generate real network activity, or carry out any real attack. Commands are represented as **text rows in a CSV only**.

### Supported categories

#### 1. `ransomware`
- **Malicious behavior:** mass file archiving, encryption-like behavior, or destructive preparation activity hidden among legitimate backup and maintenance commands.
- **Benign lookalike:** standard system backups, log rotation, archive creation, compression jobs, and storage maintenance.

#### 2. `lateral_movement`
- **Malicious behavior:** remote process execution patterns or remote administration-like commands that simulate movement between systems.
- **Benign lookalike:** legal IT mass deployments, remote software installation, enterprise administration, and fleet management tasks.

#### 3. `persistence`
- **Malicious behavior:** creation or modification of scheduled tasks, services, startup items, cron jobs, or other mechanisms that simulate attacker persistence.
- **Benign lookalike:** legal software updates, monitoring agents, backup cron jobs, package maintenance, and recurring administrative tasks.

#### 4. `credential_dumping`
- **Malicious behavior:** suspicious copying or access to sensitive authentication-related files, registry-like locations, browser databases, or credential storage artifacts.
- **Benign lookalike:** standard administrative backup tasks, migration scripts, profile backup jobs, system state backups, and compliance inventory collection.

#### 5. `reverse_shell`
- **Malicious behavior:** suspicious interactive shell behavior, shell pipes, redirection, or encoded command execution that simulates attacker-controlled command execution.
- **Benign lookalike:** legitimate automated DevOps remoting, CI/CD shell automation, remote health checks, and scripted maintenance sessions.

#### 6. `data_exfiltration`
- **Malicious behavior:** collection, compression, staging, or simulated upload of user, application, or log data.
- **Benign lookalike:** routine log syncing, backup uploads, observability exports, support bundles, and application diagnostics collection.

#### 7. `sql_exploitation`
- **Malicious behavior:** suspicious operating-system commands spawned in a database context, or commands that simulate abuse of database maintenance features.
- **Benign lookalike:** heavy database routine maintenance, backups, migrations, indexing jobs, exports, imports, and administrative scripts.

#### 8. `crypto_miner`
- **Malicious behavior:** background heavy-compute activity hidden under generic process names, long-running loops, or suspicious worker-like execution.
- **Benign lookalike:** QA stress testing, performance benchmarks, load testing, build jobs, analytics processing, and legitimate high-CPU maintenance.

#### 9. `privilege_escalation`
- **Malicious behavior:** commands that simulate attempts to gain elevated permissions, abuse service configuration, use administrative override mechanisms, or exploit weak local configuration.
- **Benign lookalike:** legal administrative overrides, service repair, permission fixes, emergency maintenance, and approved IT troubleshooting.

#### 10. `defense_evasion`
- **Malicious behavior:** commands that simulate altering logs, security controls, firewall settings, service configuration, or monitoring behavior to avoid detection.
- **Benign lookalike:** corporate-wide official security policy updates, firewall rule deployments, logging configuration changes, endpoint management, and authorized hardening tasks.

### Design rules

- **Pair malicious categories with their benign lookalikes.** For every malicious category the user selects, the dataset composer should try to include the related benign lookalike categories in the 200 benign rows. This makes the dataset more realistic and prevents the Blue Team from relying on simple process-name or keyword matching.
- **The CSV generator receives counts, not command text.** The generator should receive category counts (not raw command strings), pull matching command rows from the command database, validate the final label split, and produce the final CSV files.

## 12. Example Request Shape

In plain English, the CSV generator should expect a request that contains:

- **A story or scenario** — the attack context the judges asked for (e.g. "ransomware on a Linux web server").
- **An OS profile** — e.g. `linux` or `windows`.
- **A dictionary of benign category counts** whose values **total 200** — e.g. so many `devops`, so many `logs`, so many `backup`, and so on across the benign categories.
- **A dictionary of malicious category counts** whose values **total 20** — e.g. a few `discovery`, a few `persistence`, a few `impact`, distributed across the malicious categories to fit the scenario.

The generator uses these counts to pull the right commands from the database, validate the totals, blend the rows, and write the output files.
